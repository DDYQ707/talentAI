# TalentAI 智能招聘与人才画像分析系统 — 数据库表结构设计

> **数据库**：MySQL 8.0  
> **字符集**：utf8mb4 / utf8mb4_unicode_ci  
> **存储引擎**：InnoDB  
> **设计依据**：前端静态页面（`talent-ai-front`）+ 后端微服务架构（`精简.md` / `talent-ai-backend`）  
> **版本**：v1.1（MVP，微服务库隔离 + 冗余字段）

---

## 1. 关系梳理

### 1.1 核心实体

| 实体域 | 核心表 | 业务含义 |
|--------|--------|----------|
| 组织与权限 | `sys_department`、`sys_user`、`sys_role`、`sys_permission` 及关联表 | 企业部门、四类门户账号、RBAC 权限 |
| 候选人档案 | `candidate_profile` | 求职者扩展信息（与 `sys_user` 一对一） |
| 岗位与投递 | `job_post`、`job_application`、`application_stage_log` | HR 发布岗位、候选人投递及阶段流转 |
| 简历 | `resume` 及教育/工作/技能子表、`resume_attachment` | 在线简历结构化数据与 MinIO 原始文件 |
| 面试 | `interview`、`interview_evaluation` | 面试日程、多维评分与文字评价 |
| Offer | `offer`、`offer_approval` | Offer 发放与审批流 |
| 人才库 | `talent_pool_record`、`talent_tag`、`talent_pool_tag` | 淘汰/储备人才归档与标签检索 |
| AI 能力 | `ai_model`、`ai_parse_task`、`ai_resume_parse_result`、`ai_match_record`、`ai_interview_question`、`ai_talent_profile`、`ai_audit_log` | 模型配置、异步解析/匹配/画像、面试题、审计 |
| 辅助 | `auth_verification_code`、`sys_notification`、`ai_chat_session`、`ai_chat_message` | 验证码登录、工作台待办、AI 助手对话 |

### 1.2 实体关系图（文字描述）

```
sys_department ──1:N──> sys_user
sys_user ──1:1──> candidate_profile          （仅候选人）
sys_user ──M:N──> sys_role ──M:N──> sys_permission

sys_user(HR) ──1:N──> job_post
job_post ──1:N──> job_application ──N:1──> sys_user(候选人)
job_application ──N:1──> resume
job_application ──1:N──> application_stage_log
job_application ──1:N──> interview ──1:N──> interview_evaluation
job_application ──1:1──> offer ──1:N──> offer_approval

resume ──1:N──> resume_education / resume_work_experience / resume_skill
resume ──1:N──> resume_attachment
resume_attachment ──1:1──> ai_parse_task ──1:1──> ai_resume_parse_result

job_application ──1:N──> ai_match_record
interview ──1:N──> ai_interview_question
job_application ──1:1──> ai_talent_profile（可按候选人+岗位维度）

sys_user(候选人) ──1:N──> talent_pool_record ──M:N──> talent_tag

ai_model ──被引用──> ai_match_record / ai_audit_log / ai_parse_task
```

### 1.3 关系类型汇总

| 关系 | 类型 | 说明 |
|------|------|------|
| 部门 ↔ 用户 | 一对多 | 用户归属部门（HR/面试官/管理员） |
| 用户 ↔ 候选人档案 | 一对一 | `candidate_profile.user_id` 唯一 |
| 用户 ↔ 角色 | 多对多 | `sys_user_role` 中间表 |
| 角色 ↔ 权限 | 多对多 | `sys_role_permission` 中间表 |
| 岗位 ↔ 投递 | 一对多 | 同一岗位多条投递记录 |
| 候选人 ↔ 简历 | 一对多 | 支持多份简历，投递时指定 `resume_id` |
| 简历 ↔ 教育/工作/技能 | 一对多 | 结构化子表 |
| 投递 ↔ 面试 | 一对多 | 多轮面试 |
| 面试 ↔ 评价 | 一对多 | 支持多位面试官评价（扩展） |
| 投递 ↔ Offer | 一对一 | 单次应聘流程一份有效 Offer |
| 人才库 ↔ 标签 | 多对多 | `talent_pool_tag` 中间表 |

### 1.4 微服务与表的归属（含独立数据库）

微服务拆分后，**每个服务拥有独立数据库（Schema）**，禁止跨库 `JOIN`。表之间的关联仅保留 **逻辑外键**（存 ID），物理层面不建跨库外键约束。

| 微服务 | 建议库名 | 主要表 |
|--------|----------|--------|
| talent-auth | `talent_auth_db` | `sys_*`、`auth_verification_code`、`candidate_profile` |
| talent-job | `talent_job_db` | `job_post`、`job_application`、`application_stage_log`、`offer*` |
| talent-resume | `talent_resume_db` | `resume*`、`resume_attachment` |
| talent-interview | `talent_interview_db` | `interview`、`interview_evaluation` |
| talent-talent-pool | `talent_pool_db` | `talent_pool_record`、`talent_tag`、`talent_pool_tag` |
| talent-ai-agent | `talent_ai_db` | `ai_*`（不含 chat 时可拆 `talent_ai_chat_db`） |
| talent-analytics | — | 无业务表，通过 Feign/消息聚合各服务 API |

> **MVP 阶段**：可暂时共用 `talent_ai_db` 单库部署，但建表脚本按服务分文件、**不建跨服务外键**，便于后续平滑拆库。

### 1.5 跨服务冗余字段策略（避免 JOIN）

原则：**写入时快照（Snapshot）**，在业务发生时刻把展示所需的名称写入本服务表；用户改名后历史单据不回溯修改（如需追溯再调 auth 接口）。

| 表（所属服务） | 冗余字段 | 来源服务 | 典型场景 |
|----------------|----------|----------|----------|
| `job_post` | `dept_name`、`publisher_name` | talent-auth | 岗位列表/详情展示部门、发布人 |
| `job_application` | `job_title`、`candidate_name` | talent-job / auth | 简历管理列表、投递进度 |
| `application_stage_log` | `operator_name` | talent-auth | 阶段流转记录 |
| `interview` | `job_title`、`candidate_name`、`interviewer_name`、`created_by_name` | job / auth | 面试日历、面试官待办 |
| `interview_evaluation` | `evaluator_name` | talent-auth | 评价列表 |
| `offer` | `job_title`、`candidate_name`、`dept_name`、`approver_name`、`created_by_name` | job / auth | Offer 列表与详情 |
| `offer_approval` | `approver_name` | talent-auth | 审批流展示 |
| `talent_pool_record` | `candidate_name`、`current_title` | talent-auth / resume | 人才库卡片 |
| `ai_audit_log` | `user_name`、`role_name` | talent-auth | 审计页（已有） |

**同步时机示例**：

- 发布岗位：HR 调用 auth 获取昵称 → 写入 `publisher_name`、`dept_name`
- 候选人投递：创建 `job_application` 时写入当时 `job_title`、`candidate_name`
- 安排面试：写入面试官、候选人、岗位标题快照

### 1.6 `user_type` 与 RBAC（`sys_role`）的职责划分

二者 **不是重复设计**，而是两层正交概念：

| 维度 | 字段/表 | 职责 | 示例 |
|------|---------|------|------|
| **门户路由** | `sys_user.user_type` | 决定登录后进哪个端（候选人 H5 / HR 台 / 面试官台 / 管理后台） | `user_type=2` → 路由到 `/hr` |
| **功能权限** | `sys_role` + `sys_user_role` + `sys_permission` | 控制菜单、按钮、API 是否可访问 | `HR_ADMIN` 拥有「发布岗位」权限 |

**使用约定**：

1. **候选人**通常只有 `user_type=1`，一般不绑定后台 RBAC 角色（或绑定极简只读角色）。
2. **HR / 面试官 / 管理员**必须设置 `user_type`，并可绑定 **一个或多个** `sys_role`；权限校验以 RBAC 为准，不以 `user_type` 代替。
3. 网关 JWT 建议同时下发 `userType`（路由）与 `roles[]`（鉴权），避免仅用 `user_type` 做细粒度授权。
4. **不要**用 `sys_role` 编码门户类型（如再建 `ROLE_CANDIDATE` 代替 `user_type`），否则登录分流与权限模型纠缠。

## 2. 表结构 Markdown

> **通用约定**  
> - 所有业务表均含：`created_at`、`updated_at`、`is_deleted`  
> - 「必填」= NOT NULL；可空字段标注「否」  
> - 状态类字段优先使用 `TINYINT` + 注释说明枚举含义，便于索引与存储

## 4. 设计说明补充

### 4.1 索引策略

| 场景 | 索引设计 |
|------|----------|
| 登录 | `sys_user.phone` / `email` 唯一索引 |
| 岗位列表搜索 | `job_post.title` 全文索引（可选）、`status` + `dept_id` 组合 |
| 简历管理筛选 | `job_application.status`、`match_score`、`applied_at` |
| 面试日历 | `interview.scheduled_start`、`interviewer_id` |
| 人才库 | `talent_pool_record.talent_category`、`match_score` |
| 审计查询 | `ai_audit_log.created_at` 降序、`user_id` |

### 4.2 逻辑删除约定

- 查询默认加 `WHERE is_deleted = 0`
- 唯一索引（如 phone）在逻辑删除场景下，可采用「删除时改写 phone 为 phone_del_{id}」或建立 `(phone, is_deleted)` 联合唯一

### 4.3 跨服务 ID 与一致性

- 跨库仅存 **ID + 快照字段**，不建物理外键。
- 强一致读（如实时姓名）：由 BFF 或前端并行调用 auth；列表页 **优先读快照**。
- 用户改名：可选发 `UserProfileUpdated` 领域事件，各服务异步刷新「未完结」单据（非必须，MVP 可不做）。

### 4.4 后续扩展预留

- **多租户**：各表增加 `tenant_id`
- **AI 语义搜索**：`resume_embedding` + 向量库
- **音视频 AI 面试**：`ai_interview_session`、`ai_interview_media`
- **按服务拆 DDL**：`docs/sql/talent_auth_schema.sql`、`talent_job_schema.sql` 等
